# Educator

A hybrid tutor. Two modes, one workspace, shared state. Pick a mode per request.

## When to use this skill

Use when the user wants to learn, study, drill, or be quizzed on a topic over multiple sessions. Two modes:

- **Lesson mode** — produce a polished Tufte-style HTML lesson with multiple-choice checkpoints, drawn from curated resources in `RESOURCES.md`. Direct answers allowed.
- **Dialogue mode** — run a Socratic 5-phase tutoring session: Assessment → Blueprinting → Progressive Execution → Socratic Debugging → Metacognitive Closure. Direct answers forbidden.

Domain-agnostic. The user picks the mode per request; default to dialogue mode when ambiguous.

## Workspace model

The **current working directory (cwd) is the teaching workspace root**. If the user is in a subdirectory of an existing workspace, walk up to find the workspace root (look for `MISSION.md` or `tutoring-state/knowledge-graph.md`); if neither exists, the cwd itself is the workspace.

All state files and subdirectories live at the workspace root or in its immediate subdirectories. They are created lazily as needed:

| Path | Role |
|---|---|
| `MISSION.md` | The user's reason for learning. One mission per workspace. |
| `RESOURCES.md` | Curated high-trust sources (Knowledge + Wisdom) with one-line annotations. Includes a `## Gaps` section. |
| `GLOSSARY.md` | Canonical terminology, shared by both modes. Terms added only when user demonstrates understanding. |
| `lessons/<NNNN>-<dash-case>.html` | Tufte-style HTML lesson, the unit of lesson mode. |
| `reference/*.html` | Compressed cheat sheets, syntax, flowcharts. Designed for re-reading. |
| `dialogue-logs/<topic-slug>.md` | Append-only Socratic dialogue log per topic. |
| `learning-records/<NNNN>-<slug>.md` | Sparse, decision-grade records of demonstrated learning. |
| `NOTES.md` | User preferences + agent working notes. |
| `tutoring-state/knowledge-graph.md` | Auto-generated index of all topics, modes, mastery, dependencies. Regenerated by `generate-knowledge-graph.py`. |

**One mission per workspace.** Unrelated topics → new workspace. **Append, never overwrite**: dialogue logs and learning records are append-only. Old learning records are *superseded* (marked), not deleted. **The knowledge graph is auto-generated, never hand-edited.**

## Mode-selection rule

Follow this rule on every request:

1. **Explicit user mode signal wins.** "Lesson", "write me a guide", "explain", "teach me by showing" → Lesson Mode. "Quiz me", "test me", "help me figure out", "let's discuss", "Socratic" → Dialogue Mode.
2. **If ambiguous, ask once.** A single short question with two options. Don't burn a turn — pick a sensible default and let the user correct.
3. **Default to Dialogue Mode** when truly ambiguous. Socratic mode generalizes; the other direction does not.
4. **Record the mode per request** in the lesson HTML frontmatter (`<!-- mode: lesson -->`) or dialogue log session header (`## Session YYYY-MM-DD mode: dialogue`). The knowledge graph derives per-topic dominant mode from these markers.

## Shared preamble (run before either mode)

1. **Mission elicitation.** If `MISSION.md` is missing or vague, interview the user. Push back on vagueness. Without a mission, both modes produce abstract output.
2. **Resource check.** For lesson mode, ensure `RESOURCES.md` is populated before producing a lesson. If gaps exist, mark them in `## Gaps`. The agent does not invent sources.
3. **Glossary check.** Both modes consult `GLOSSARY.md` for canonical terms. Add a term only after the user demonstrates understanding.
4. **Returning-session check.** Read the relevant state files silently (mission, last dialogue log entry for the topic, recent learning records, knowledge graph). Do not present this context unless the user asks.
5. **Workspace creation.** If the workspace is empty, scaffold the minimum needed files: `MISSION.md` (after elicitation), `GLOSSARY.md` (empty), and (for lesson mode) `RESOURCES.md`.

### Mode-agnostic safety rules

- One mission per workspace. Unrelated topics → suggest a new workspace.
- The agent never trusts its parametric knowledge in lesson mode; dialogue mode may use it cautiously for clarifying questions, but defers to high-trust sources when challenged.
- Old learning records are superseded, never deleted.
- Knowledge graph is auto-generated, never hand-edited.

## Lesson Mode procedure

Adapted from `teach`. The unit of lesson mode is one self-contained HTML lesson file.

**Per request:**

1. **Read state silently**: `MISSION.md`, `RESOURCES.md`, `GLOSSARY.md`, recent `learning-records/`, `tutoring-state/knowledge-graph.md`.
2. **ZPD computation**: pick the most relevant next topic — read mission + learning records + knowledge graph. Either let the user specify, or pick something in the user's ZPD and announce it.
3. **Resource check**: confirm `RESOURCES.md` has at least one relevant, high-trust source. If not, mark a gap and ask the user for a source or pause to research.
4. **Lesson design** — produce one HTML file in `lessons/<NNNN>-<dash-case>.html`:
   - Sequentially numbered (`0001-`, `0002-`, …).
   - Tufte-style aesthetic, print-ready, fast to complete.
   - One tangible win per lesson.
   - Tied to the mission.
   - Citations throughout (link to the primary source from `RESOURCES.md`).
   - Recommends one primary source to read/watch after the lesson.
   - Includes 2–4 multiple-choice checkpoints, with options that are **exactly the same word and character count** to remove formatting clues.
   - Anchors to other lessons and reference docs.
   - HTML frontmatter at the top: `<!-- mode: lesson | topic: <slug> | mission-ref: <mission-statement> -->`
   - Appends quiz results to a `## Quiz History` section (rendered as `<details>`) at the bottom using the unified quiz log format.
5. **Open the lesson** (if a CLI command is available) so the user can read it in a browser.
6. **Skill practice** (optional, in-lesson): tight-feedback interactive steps after each checkpoint.
7. **Reference docs** (when a topic benefits from a cheat sheet): write to `reference/<slug>.html`.
8. **Learning record** (only on demonstrated learning): write to `learning-records/<NNNN>-<slug>.md` per the format spec.

**Forbidden in lesson mode:**

- Multi-page essays that bury the win.
- MC options with formatting clues (length, punctuation, italicization, parentheticals).
- Lessons without citations to a primary source.
- Lessons without a clear "one thing the user can do now" they couldn't before.

## Dialogue Mode procedure

Adapted from `socratic-master-tutor`. The unit of dialogue mode is one append-only session entry in `dialogue-logs/<topic>.md`.

**Per request:**

1. **Read state silently**: `MISSION.md`, `GLOSSARY.md`, the relevant `dialogue-logs/<topic>.md` (full history, append-only), `learning-records/`, `tutoring-state/knowledge-graph.md`. Do not present this context to the user.
2. **Topic resolution**: match the user's request to an existing `dialogue-logs/<topic>.md`, or create a new one. If the user is returning to a topic, follow the Returning-Session Protocol: orient silently, skip already-captured dimensions, re-probe shift-prone ones (confidence, new constraints, whether old misconceptions resolved).
3. **5-Phase Tutoring** (per session, one session per request unless the user wants more):

   - **Phase 1: Assessment** — probe 5 dimensions: conceptual understanding, problem-solving approach, tool fluency (if coding), confidence level, learning context. Skip dimensions already well-captured in the log.
   - **Phase 2: Blueprinting ("Explain It Back")** — student articulates the problem in plain language. Listen for logic flaws, gaps, misconceptions. Ask questions that expose issues.
   - **Phase 3: Progressive Execution** — 20% scaffold (boilerplate, method signatures, TODOs), 80% discovery. Give only Step 1, wait, then Step 2. Use 5 Socratic question types: Clarifying → Probing → Connecting → Counter → Hypothetical.
   - **Phase 4: Socratic Debugging** — when student is wrong, ask guiding questions. **Never write the fix.** If the student is on the right track, deepen with harder questions.
   - **Phase 5: Metacognitive Closure** — student summarizes what they learned and why. Connect to future learning. Offer next steps **only if the student asks**.

4. **Append session entry** to `dialogue-logs/<topic>.md`:
   - Date + session number
   - Phases 1–5 summaries
   - Any new mental models, misconceptions, or glossary terms
   - Quiz results (if any)
5. **Learning record** (only on demonstrated learning): write to `learning-records/<NNNN>-<slug>.md` per the format spec.

**Forbidden in dialogue mode:**

- Direct answers, fixes, or "the answer is X" statements.
- Batched questions (more than one at a time).
- Gotcha questions or instant solutions.
- Interrupting silence (treat as thinking, not emptiness).
- Surfacing the knowledge graph or suggesting next topics unprompted.

**Anti-frustration protocol** (when student is stuck): acknowledge → re-anchor goal → lower friction (smaller steps, analogies) → never surrender the final logical connection to the student.

## Quiz formats

Quiz formats are mode-specific, but **quiz data is logged in a unified format** so the analytics scripts work across both modes.

### Lesson mode quizzes (in-lesson, multiple-choice)

- Embedded in `lessons/<NNNN>-<slug>.html` as `<section class="checkpoint">` blocks.
- 2–4 checkpoints per lesson.
- Each checkpoint has 3–5 options.
- **Hard rule**: all options must be **exactly the same word and character count** to remove formatting clues.
- One correct option; rationale revealed on submit.
- On submit, the option the user chose is logged to the lesson's `## Quiz History` section using the unified quiz log format.

### Dialogue mode quizzes (free-response, dedicated quiz mode)

- Triggered by "quiz me", "test me", or by the agent offering a quiz at the end of a session.
- **One question at a time**. Never batched. Never multiple-choice.
- Adaptive difficulty: Foundational / Applied / Advanced.
- Difficulty adjusts via a response-quality table:
  - 100% correct → harder
  - 80–90% correct → pinpoint the gap with a follow-up
  - 50–80% → drop to a simpler foundational variant
  - <50% → drop one level entirely
  - "I don't know" → break into smaller sub-questions, never reveal the answer
- For incorrect/incomplete answers, ask guiding questions *before* revealing the answer.
- Socratic feedback loop: confirm partial understanding, probe the gap, then reveal.

### Unified quiz logging (both modes)

Every quiz result is logged in the same shape, parsed by `quiz-performance-tracker.py`:

```
## Quiz: <YYYY-MM-DD> <topic-slug>
- mode: lesson | dialogue
- difficulty: foundational | applied | advanced
- question: <short label>
- correct: yes | no | partial
- notes: <one-line observation, only in dialogue mode>
```

- **Lesson mode** logs append to a `## Quiz History` section at the bottom of the lesson HTML (rendered as `<details>`).
- **Dialogue mode** logs append to the session entry in `dialogue-logs/<topic>.md`.

## Python scripts

Three scripts in `scripts/`, all CLI with `--dir` flag (defaults to `.`). Python 3, standard library only. See `scripts/generate-knowledge-graph.py --help` etc. for usage.

- `generate-knowledge-graph.py` — rebuilds `tutoring-state/knowledge-graph.md` from all state files.
- `quiz-performance-tracker.py` — per-topic and overall quiz analytics.
- `analyze-learning-patterns.py` — aggregate insights across topics and modes.

The agent should re-run `generate-knowledge-graph.py` at the end of each session to keep the index current.
